# Spec for Multi-Centre Neighbourhood Tiling
branch: feature/multi-centre-neighbourhood-tiling

## Summary

Replace the single Milan centre point in Stage 1 seed acquisition with a configurable list of neighbourhood anchor points. Each anchor has its own `lat`, `lon`, and `outer_radius_m`, allowing dense, targeted tile coverage around high-venue-density areas. Linear streets (e.g. Navigli, Corso Sempione) are represented as multiple anchors spaced ~500 m apart. Tiles from all anchors are merged and deduplicated before any API queries are issued. When the anchor list is empty the pipeline falls back to the existing single-centre behaviour.

## Functional Requirements

- `Settings` exposes a `neighbourhoods` field: a list of anchor objects, each with `lat: float`, `lon: float`, and `outer_radius_m: int`.
- When `neighbourhoods` is non-empty, `run_mode_list` generates tiles for every anchor using the existing `generate_tiles` function and merges the resulting lists.
- Duplicate tile centres — defined as two tiles whose centres fall within the same step-grid cell (i.e. within `tile_radius_m * (1 - overlap)` of each other) — are removed so the same geographic area is queried at most once.
- When `neighbourhoods` is empty, behaviour is identical to the current implementation (single centre from `milan_center_lat` / `milan_center_lon` with `outer_radius_m`).
- Default `neighbourhoods` in `Settings` covers the following Milan areas with appropriate anchor counts and radii:

  | Area | Anchor count | Notes |
  |---|---|---|
  | Duomo / Centro | 1 | Existing centre point |
  | Navigli | 3 | Canal corridor, anchors spaced ~500 m |
  | Brera | 1 | Compact art-district circle |
  | Isola | 1 | Compact circle |
  | Porta Venezia | 2 | Long corso, 2 anchors |
  | Porta Romana | 2 | Long corso, 2 anchors |
  | Corso Sempione | 3 | Straight strip, 3 anchors |
  | Loreto | 1 | Dense junction |

- All anchor coordinates and radii are overridable via environment variables or `.env` (Pydantic settings serialisation of list fields).
- The tile count logged at startup reflects the deduplicated total across all anchors.

## Possible Edge Cases

- Two anchors so close together that all their tiles are deduplicated — the anchor with no remaining unique tiles should produce no queries (not an error).
- An anchor placed outside the Milan metropolitan area — no guard required, the pipeline trusts the operator's config.
- Very large `outer_radius_m` on an anchor causing the merged tile list to exceed memory — acceptable risk given the scale; no limit enforced.
- `tile_overlap` close to 1.0 producing a very dense grid — deduplication must still terminate in reasonable time.
- Checkpoint state (`list_tiles.json`) from a previous single-centre run being reused after switching to multi-centre — already-completed tiles are skipped correctly because the checkpoint key is the tile's `(lat, lon, radius_m)` tuple, independent of which anchor produced it.

## Acceptance Criteria

- Running `uv run pytest` passes with no regressions against existing Stage 1 tests.
- When `neighbourhoods` is set to the default list, the total deduplicated tile count is strictly less than `sum(tiles_per_anchor_without_dedup)` — confirming deduplication is active.
- When `neighbourhoods = []`, the pipeline produces the same tile list as before this change (regression test against a fixed seed).
- A tile that would be generated by two overlapping anchors appears exactly once in the merged list.
- The `ListReport` emitted after a run includes a `total_tiles` field that reflects the deduplicated count.

## Open Questions

- Should each anchor support its own `tile_radius_m` override, or should a single global `search_radius_m` apply to all anchors?  global.
- Should the `milan_center_lat` / `milan_center_lon` / `outer_radius_m` top-level fields be deprecated now, or kept indefinitely as the single-centre fallback? - those are defaults, so probably kept. but somewhere readable config of high denisity neigbourhoods with option to call for load on all of them or separately one by one

## Out of Scope

- Polyline / corridor geometry (Option B) — streets are approximated by multiple point anchors only.
- Per-anchor tile density or overlap settings.
- Any UI or visualisation of the tile grid.
- Automatic derivation of anchor points from OpenStreetMap or any external source.

## Feature Testing Guidelines

Create tests in `tests/stage1_seed/test_tiling.py` (extend existing file) and optionally a new `tests/stage1_seed/test_multi_centre.py`. Cover:

- `generate_multi_centre_tiles` with two non-overlapping anchors returns the union of both individual tile lists.
- `generate_multi_centre_tiles` with two identical anchors returns the same tiles as a single call (deduplication removes all duplicates).
- `generate_multi_centre_tiles` with two partially overlapping anchors returns fewer tiles than the sum of both individual counts.
- `run_mode_list` with `neighbourhoods = []` produces the same tile sequence as the current single-centre path (mock `generate_tiles` to verify call args).
- `run_mode_list` with a non-empty `neighbourhoods` list calls `generate_multi_centre_tiles` instead of the single-centre path.
